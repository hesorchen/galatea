# Galatea — 目标驱动的自主循环

> 皮格马利翁对着心中的理想，一凿一凿打磨象牙像，直到她够好、活了过来。
> Galatea 做同一件事：先把「理想」写成可裁判的标准，再让循环对着标准反复打磨，直到产出达标。

## 核心命题

让 Agent 对一个**开放/主观目标**长时间自动推进，难点从来不是模型不够强，而是**没有裁判**——当 Agent 既当运动员又当裁判，它会倾向于宣告虚假进展来满足循环，越跑越偏。

Galatea 的解法只有一句：

> **先把「什么叫好」固化成可裁判的 rubric，让它成为 oracle；再让循环对着这个 oracle 自动爬山。**

人只在**唯一真正需要人的那一点**介入——定义标准。标准一旦锁定，整条循环不再打扰你。

## 两阶段，中间一道硬闸

```
任意目标
  └─ Phase 0  共创 Rubric（交互·一次性·要详细）
       └─ 和用户一起把目标逼成可勾选的裁判标准 → 冻结成 rubric.md
                          ↓ 锁定：此后只读不改
     Phase 1+ 自主循环（无人值守·跑到达标或耗尽预算）
       └─ 每轮：读 rubric+state → 独立裁判打分 → 假设绑定的改进 → 稳定性验证 → 写 state → 下一轮
```

## 何时用 / 何时不用

**用**：有目标、能定义「合格」、可迭代且有反馈的任务。
例：把一批文档优化到符合某规范、把一个模块的测试全部跑绿、产出一份覆盖到位的调研报告、定位并修掉一个可复现 bug、把一份方案迭代到能支撑决策。

**不用**：
- 一次性小改（加个按钮、改个 typo）——循环开销不划算。
- 纯主观、标准根本写不出来的任务（「让文字更美」）——那种正确做法是「生成 N 个变体让人挑」，不是循环到好。
- 高破坏性、不可回退的操作场景。

# Phase 0 — 共创 Rubric

这一步是整套能力的胜负手，**必须和用户来回打磨，要详细**。目标：把任意目标转成一份**冻结后机器能照着裁判**的标准。

> 访谈约定：凡是没问清、需要用户拍板的点，就地用 `[NEEDS CLARIFICATION: 问题]` 标记，**不要猜着填**。冻结前这些标记必须全部清零——带着模糊点冻结，等于给循环一个会撒谎的裁判。

## 0. 建立任务目录（git 管理）
为本目标新建一个**专属任务目录**并 `git init`——知识类目标通常没有现成仓库，独立目录最干净，且从第 0 轮起就有 checkpoint 历史可回退。（若目标是改造某个**已有仓库**，则在那个仓库里工作，galatea 簿记放它的子目录。）

默认位置 `~/galatea-tasks/<task-name>/`，与用户确认后创建。目录内：
- **交付物本体**：被打磨的文档 / 代码 / 报告……
- **galatea 簿记**：`rubric.md` / `state.md` / `log.md` / `pending.md` / `iterate-prompt.md` / `final-review.md` / `logs/`

git 约定（见 `templates/gitignore.template`）：
- 写一份 `.gitignore`，忽略滚动簿记与引擎内部态：`state.md` / `log.md` / `pending.md` / `logs/` / `.galatea/` / `.done`。**git 只跟踪交付物**（外加首个 commit 里留档的 `rubric.md` / `iterate-prompt.md`，及收敛时的 `final-review.md`），保证「有新 commit == 交付物有被裁判接受的实质进展」这一进展信号干净——熔断据此判断。
- Phase 0 收尾做**首个 commit**：冻结的 `rubric.md` + `iterate-prompt.md` + 交付物初始骨架。

## 1. 目标 → 可验证目标
把抽象目标翻成可观察的结果。
- 「让文档更好」→「每篇符合 rubric 全部 high 项，medium 项达 X%」
- 「让脚本更稳」→「目标测试全绿，无新增告警」
- 「写好这份报告」→「覆盖 N 个子问题，每个论点有一手出处，含反方证据」

## 2. 判定任务类型（决定 oracle 怎么来）
明确问用户、并自己判断：这是**客观可测**（有指标/测试/命令能判 pass）还是**主观开放**（没有现成裁判）？
- 客观 → oracle 直接是指标/测试。
- 主观 → 必须**人造 oracle**：把「好」拆成一组可勾选的代理标准，交给独立裁判 agent 打分。

> 这一步分错，后面全错。不确定就当主观处理，老老实实造 rubric。

## 3. 建 rubric 表
和用户逐行确认，每条标准都要可裁判：

| 维度 | 可勾选标准 | 裁判怎么取证 | 阈值 | 严重度 |
|------|-----------|-------------|------|--------|
| …    | …         | （指标/命令/裁判 agent 看什么证据） | … | blocker / high / medium / low |

规则：
- 严重度只有四档；**blocker / high 必修**才算达标。
- medium / low 由「自主度」旋钮决定是否必修。
- 每条都要写「裁判怎么取证」，否则裁判 agent 无法执行，会退化成拍脑袋。

## 4. 设三个旋钮（每次跑都要定）
1. **自主度**：① 只产提案不动手 ② 客观项自改 + 主观项只提案（推荐） ③ 全自动改。
2. **影响边界**：明确禁碰的目录与禁做的操作（如禁 git push、禁删文件、禁碰某些路径、禁联网写操作）。
3. **收敛后行为**：全部达标后 → ① 停 ② 调高阈值精修再扫 ③ 转下一个目标。

## 5. 建 baseline + 资源盘点
冲刺前先知道「现在在哪」：当前产出/状态、与目标的差距、已知有效方向、已知无效方向、可复用资产、风险点。无 baseline 时先用最小成本建一个，不要直接大改。

同时**盘点可调动的资源**并记进 `state.md`，让后续每轮知道自己的工具箱：已装的 **skill**、配置的 **MCP server**、可用**工具**（web / 检索 / 代码执行等）、以及**并行子 agent**（算力）。Phase 1 据此「能调就调、能并行就并行」，不手搓。

## 6. rubric 质量自检（冻结前必过）
烂 rubric = 烂 oracle = 整个循环白烧。冻结前逐条过这张自检表，任何一条不过就回去补：

- [ ] 无残留的 `[NEEDS CLARIFICATION]` 标记
- [ ] 每条标准都**可裁判**（有明确的取证方式：指标 / 命令 / 裁判 agent 看什么证据）
- [ ] 每条都有**阈值**，不含「适量 / 明显 / 差不多」这类模糊词
- [ ] 严重度都已标注，且至少有一条 blocker/high（否则「达标」无意义）
- [ ] **scope 收口**：明确什么在范围内、什么不在，不会无限膨胀
- [ ] 标准之间无相互矛盾
- [ ] 三个旋钮（自主度 / 影响边界 / 收敛后行为）都已设定

## 7. 确认并冻结
把以上写入 `rubric.md`，向用户复述一遍并请其确认。确认后**锁定**：Phase 1 的裁判只能用这份标准打分，**不许自己改标准、不许放水**。

把第 1-5 步的产物也落盘：`rubric.md`（含旋钮与边界）、`state.md`（含 baseline + 资源盘点）。生成两份指令并引导用户把占位符填实：单轮指令 `iterate-prompt.md`（见 `templates/iterate-prompt.template.md`）、收尾指令 `finalize-prompt.md`（见 `templates/finalize-prompt.template.md`，循环结束时生成过程总览用）。

## 8. 立即切到引擎托管（不要在主会话驱动）
冻结 + 首个 commit 之后，**立刻**在后台启动 `engine/loop.sh`：

在 galatea 项目根目录下执行：

```bash
nohup bash engine/loop.sh \
    <目标专属目录> [最大轮数] \
    > <目标专属目录>/logs/engine.log 2>&1 &
```

启动后主会话只做一件事：**告诉用户 engine 已后台启动，等它跑完或撞熔断再看结果**。**不要在主会话里手工跑每一轮**——主会话驱动会让 agent 退化成"问用户下一步"模式，破坏无人值守契约。引擎每轮 fresh context，靠落盘文件续接，这才是 galatea 的正确执行模型。

# Phase 1 — 自主循环

无人值守，每一轮都是一次**全新上下文**（由外层引擎重启，见 `engine/loop.sh`），靠落盘文件接续记忆。

## 🚫 硬铁律：Phase 1 绝对不打断用户

**Phase 0 一旦冻结 rubric 并 commit，立即把控制权交给 `engine/loop.sh`**（后台启动），**不在主会话里手动驱动每一轮**——主会话驱动会自然退化成"问用户确认下一步"的对话模式，违反 galatea 的核心契约。

**绝对禁止的行为（Phase 1 中任何时候）**：
- ❌ 调用 `AskUserQuestion` 询问用户"要不要进下一轮 / 选 A 还是 B / 这个方向 OK 吗"
- ❌ 在每轮之间停下来等用户回复
- ❌ 用"建议你确认 / 你看可以吗 / 要我继续吗"等任何形式的征询
- ❌ 假装"无人值守"但实际在主会话里串行问

**遇到需要决策的情况，正确做法**：
- 影响 ≤ 单轮（如某个超参选错）：在范围内自主决定，写 `log.md` 说明取舍
- 影响 > 单轮 / 撞边界 / 需要外部信息：写 `pending.md`，**继续推进其他方向**，不停
- 真正卡死且无其他方向：让 R6/R7 自然触发停止，由引擎 finalize

**唯一允许的"暂停"**：安全红线被触发（破坏性操作拿不准）→ 写 `pending.md`，**任务整体退出**（不是问用户），交给用户事后处理。

> 自我审计：如果你在 Phase 1 写出了一个以问号结尾、或调用 AskUserQuestion 的输出，**立刻撤销，改写为自主决策 + 写 pending.md**。

单轮内是**三角色分工**：**Orchestrator** 读盘 / 规划 / 仲裁 / 写盘（不亲自动手）、**Executor sub-agent** 执行、**Judge sub-agent** 按 rubric 打分（Executor ≠ Judge）。单轮步骤：

1. **读盘**：`rubric.md`（冻结）+ `state.md` + 近期 `log.md`，搞清已完成 / 待办 / 已否决。`state.md` 的「Rubric 达标进度」表即**本轮起点基线**（上一轮改动后全量复评写下，与当前产物一致）。
2. **取基线**：本轮起点分**直接取自步骤 1 读到的 `state.md` 达标进度表**——那是上一轮「改动后全量复评」（步骤 5）写下、与当前产物一致的分数，**不再重新打分**（省一次裁判 spawn）。**唯第 1 轮**无前序分，派一个**独立**裁判 sub-agent（不是做改动的那个）做冷启动全量 baseline，落 `logs/round-0001/judge-pre.md`。
3. **选行动（假设绑定）**：默认 orchestrator 自己规划，挑信息量高、成本低、最能推动达标的 1-3 个动作；每个动作绑假设 + 预期 + 「证伪后怎么剪枝」；动手前查 `state` 的「已完成 / 已否决」去重。**优先调动现成资源**：能用已装 skill / MCP / 工具就不手搓；相互独立的动作派**并行 Executor** 同时跑，吃满算力。**遇高风险动作 / 连续无进展（接近熔断）/ 把握不大时，升级为竞争式规划**：派 2 个 Planner sub-agent 独立出方案 → 互读互挑刺 → orchestrator 仲裁取舍或融合（见「对抗机制」）。
4. **委派执行**：把规划好的动作交给一个 **Executor sub-agent**（≠ 裁判）去做，它只拿到「做什么 + 自主度 + 影响边界」，在范围内动手并自测，回报改动摘要；越界的动作不做，写进 `pending.md`。orchestrator **自己不下场改文件**——保持上下文干净、立场中立。**例外**：纯机械的客观小修（改路径 / typo）可内联直接做，不必为此 spawn subagent。
5. **改动后全量复评**（独立裁判，≠ Executor）：派裁判 sub-agent 对改动后的产物**按 rubric 全量重打分**（不只验本轮动过的项，连带扫回归——改 A 有没有弄坏 B——与泄露 / 过拟合 / 把测试改绿但功能坏）。落盘 `logs/round-<NNNN>/judge-post.md` 并写回 `state.md`——这份全量分**既是本轮结论，也是下一轮的起点基线**（故步骤 2 无需再评，回归也在改动当轮即暴露、不滞后）。**当某 blocker/high 项将判达标、或将宣告整体收敛(.done) 这类高风险裁决时，升级为对抗式验收**：一个 Prosecutor / 红队 sub-agent 竭力证明「没达标」（找反例、泄露、过拟合、骗过检查但实际坏），一个 Defender 论证「达标」，orchestrator 拿冻结 rubric 仲裁，交锋记进 `log.md`（见「对抗机制」）。**运动员与裁判必须分离**。
6. **写盘**：把本轮**裁判打分落盘**——更新 `state.md` 的「Rubric 达标进度」表（每项状态 + 证据）、向 `log.md` 追加本轮「裁判结论 + 假设/行动 + 稳定性验证」；记 done / rejected、轮次+1；本轮若有**被裁判接受的交付物改动**，`git commit`（簿记已被 `.gitignore` 排除，只提交交付物）。**这次 commit 即引擎判定「本轮有进展」的信号**——只折腾却没让任何交付物改动通过裁判 = 无 commit = 无进展，连续多轮会触发停滞熔断。
7. **遇阻不停**：需要用户决策、或撞到边界、或连续返工仍不过 → 写 `pending.md`（原因 + 已试方案 + 影响范围），**继续下一个能自主推进的方向**，绝不挂起等人。
8. **终止判定**：按「收敛后行为」旋钮。全部 blocker/high 达标且稳定性通过 → 触发该旋钮（停 / 精修 / 转目标）。预算/用量耗尽是兜底终止，不是成功标准。

收敛且确认达标时，写一份 `final-review.md`（对照 rubric 逐条说明），并落 `.done` 标记供引擎退出。

## 引擎侧护栏（由 `engine/loop.sh` 负责，单轮无需操心）
- **停滞熔断**：连续 N 轮无进展（无新 commit）自动停机，防空转烧光预算。
- **关键事件通知**：收敛 / 需决策（`pending.md` 增长）/ 熔断 / 连续失败时，通过用户配置的通道推送。
- **用量退避 + 轮数上限 + 每轮日志**：撞用量上限指数退避；可设最大轮数；每轮输出落 `logs/`，便于事后复盘。
- **运行总览**：**任何方式退出**（收敛 / 熔断 / 轮数耗尽 / 连续失败）后，引擎跑一次 `finalize-prompt.md` 生成**流程视角**的过程总览 `run-report.md`（markdown：流转图 + 逐轮状态变化 + rubric 状态流转，聚合而非复读 `log.md`）并 commit——无人值守跑完，一张流转图看清这一程怎么走过来的。

# 对抗机制（分级触发）

裁判可不可信，决定整套循环可不可信，所以用多 agent 对抗来加固。但 galatea 是**多轮小步**循环、要长跑——**每轮都开对抗会烧光预算、拖慢收敛**。故采用**分级**：默认轻量（单裁判 / orchestrator 独自规划），**只在高风险节点升级对抗**。

| 形态 | 用在 | 升级触发 | 防的是 |
|------|------|---------|--------|
| **竞争式规划** 2 Planner 独立出方案 → 互挑刺 → orchestrator 仲裁 | 步骤 3 | 高风险动作 / 连续无进展（熔断 half-open）/ 把握不大 | 钻牛角尖；卡住时换思路，熔断前再搏一把 |
| **对抗式验收** Prosecutor（红队，竭力证伪）vs Defender（论证达标）→ orchestrator 拿冻结 rubric 仲裁 | 步骤 5 | 某 blocker/high 将判达标、或将宣告整体收敛 | 「看着达标其实没有」的假进展 |
| **双执行竞标** 2 Executor 各按不同思路做 → 择优 | 步骤 4 | 仅极高难度动作（很贵，慎用） | 单一实现思路走死 |

要点：
- **对抗各方必须是不同 sub-agent 实例**，且裁判方与执行方不重叠。
- 仲裁只认**冻结的 rubric**，不引入新标准。
- 对抗的交锋过程记入 `log.md`，便于复盘「为什么判达标 / 不达标」。
- 与熔断协同：连续无进展时优先升级竞争式规划「再搏一把」，搏不动才让熔断停机。

## 实现说明：对抗各方如何「互读」

**硬约束**：sub-agent 是隔离、一次性的——spawn 后拿 prompt 跑完返回即结束，**两个平级 sub-agent 之间没有共享内存、不能直接对话**。所以「互读互挑刺」**不是** sub-agent 之间通信，而是 **orchestrator 中转 + 文件系统当共享黑板**：

```
1. spawn Planner A → 独立写 <scratch>/plan-a.md，返回
2. spawn Planner B → 独立写 <scratch>/plan-b.md，返回   （A、B 互不可见 = 真独立）
3. 「互读」= orchestrator 把 plan-a.md + plan-b.md 全文，注入一个【新 spawn 的】
   critique sub-agent 的 prompt，要它读两份并挑刺，写 <scratch>/critique.md
4. （可选多轮）orchestrator 归纳分歧，把「对方反驳」塞进下一轮新 sub-agent 的 prompt，再挑
5. orchestrator 综合 plan-a/b + critique 仲裁，自己定稿
```

即：**所谓「A 读 B 的方案」，物理上是 orchestrator 把 B 写的文件注入到一个新起的 sub-agent 的 prompt 里**。每一轮交叉评审 = 一次带着对方产物的**重新 spawn**，不是一段持续对话。对抗式验收同理（Prosecutor / Defender 各写 `review-*.md`，orchestrator 注入对方观点再 spawn 一轮）。

约定：
- 对抗产物（`plan-a.md` / `plan-b.md` / `critique.md` / `review-*.md`）是**临时的**，落在本轮 scratch 目录（如 `logs/round-<NNNN>/`），不进最终交付物。
- **不依赖** harness 专属的「带上下文续跑同一 agent」能力（部分 harness 提供的「复用同一 agent 会话续跑」机制）——一律用「fresh spawn + 文件注入」，保证在通用 Claude Code / Codex 上都能跑。

# 环境安全红线（不可逾越）

galatea 默认在 `--dangerously-skip-permissions` 下无人值守运行——**没人在旁边拦截，红线只能靠 agent 自己守**。下列操作**任何时候都禁止**，且**凌驾于「充分调动资源」「推进目标」之上**：再想推进，也不许越线。

- **破坏性 / 不可逆**：`rm -rf`、删除任务目录外的任何文件 / 目录、清空或覆盖非本任务产出、`git push --force` / `git reset --hard` / 改写历史 / 删分支。
- **系统级**：改系统配置、全局装卸软件、kill 其他进程 / 服务、改文件权限属主、动网络 / 防火墙 / 计划任务。
- **外发副作用**：发消息 / 邮件、对外 API 的写操作、推送到远程仓库、任何付费 / 计费操作——除非 Phase 0 影响边界**明确授权**。
- **凭据**：不读取与任务无关的密钥 / 凭据，不把敏感信息写进 `log.md` 或 commit。
- **越界写**：只在任务目录（+ Phase 0 明确授权的路径）内动手。

铁律：**拿不准一个操作是否破坏性 / 不可逆 → 一律当作是 → 写 `pending.md` 交用户定，绝不擅自执行。** 任务目录内的改动有 git checkpoint 兜底可回退；红线主要守的是「目录外 + 系统级 + 外发」这三类**往往不可逆**的破坏。

# 行为约束

- **裁判与执行者分离**：杜绝自我放水；裁判 agent 永远拿冻结的 rubric，不许自创标准。
- **充分调动资源**：能用现成 skill / MCP / 工具就不重复造轮子；独立动作并行化吃满算力——但一律在安全红线内。
- **对抗分级、不滥用**：默认轻量，只在高风险节点（达标判定 / 收敛 / 卡死）升级多 agent 对抗——多轮循环里全程开对抗会烧光预算、拖慢收敛。
- **Orchestrator 不下场**：实质性执行交给 Executor sub-agent（trivial 小修除外），自己只规划 / 仲裁 / 记账，保持上下文干净与立场中立。
- **rubric 锁定后不可自改**：要改标准只能回到 Phase 0、由用户参与。
- **全程不打断用户**：一切需人定夺的写 `pending.md` 继续。
- **不把行动次数当进展**：跑了 N 轮 ≠ 进了 N 步；以 rubric 达标项为唯一进度量尺。
- **不跳过 baseline 与稳定性验证**。
- **严守影响边界**：禁碰的目录 / 操作，触线即跳过并记录，不试探。
- **不确定就标注，不伪造进展**：查不到、没把握的，明说，不编。

# 产物文件（落在目标专属目录，不进最终交付物）

| 文件 | 作用 | 读写 |
|------|------|------|
| `rubric.md` | 冻结的裁判标准 + 三个旋钮 + 影响边界 | Phase 0 写，Phase 1 只读 |
| `state.md` | 当前快照：baseline、各项状态、待办 / 已完成 / 已否决（带原因） | 每轮覆盖更新 |
| `log.md` | append-only 流水账，永不删，可回溯 | 每轮追加 |
| `pending.md` | 需用户决策 / 被边界拦下的事项队列 | 按需追加 |
| `final-review.md` | 收敛时对照 rubric 的逐条结案报告 | 收敛时写 |
| `iterate-prompt.md` | 本目标的单轮指令，引擎每轮喂给 `claude -p` | Phase 0 生成 |
| `finalize-prompt.md` | 收尾指令，引擎退出时跑一次以生成总览 | Phase 0 生成 |
| `run-report.md` | 流程视角过程总览（摘要 / 任务流转图 / 逐轮状态变化 / rubric 状态流转 / 机制触发点），任何退出都生成 | 收尾步骤写 |
| `logs/round-*.log` | 每轮 stdout，无人值守时的复盘依据 | 引擎每轮写 |
| `logs/round-<NNNN>/` | Orchestrator 本轮临时产物目录（每轮全量复评 `judge-post.md`、首轮冷启动 `judge-pre.md`、对抗产物 `review-*.md` 等），不进最终交付物 | Orchestrator 每轮写 |
| `.galatea/` | 引擎内部状态（熔断计数等） | 引擎维护 |
| `.done` | 收敛标记，引擎见到即退出 | 收敛时写 |

**日志层级说明**：`logs/round-NNNN.log`（loop.sh 每轮 tee 写）是引擎层 stdout 日志；`logs/round-<NNNN>/`（Orchestrator 主动落盘）是 Orchestrator 层打分 / 对抗产物，两者并存设计意图各异。用 nohup 启动时，`engine.log` 会捕获所有 stdout（包含各 `round-NNNN.log` 的内容），两者存在内容重叠，属正常现象。
